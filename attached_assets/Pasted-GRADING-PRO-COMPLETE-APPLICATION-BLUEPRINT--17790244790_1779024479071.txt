GRADING PRO - COMPLETE APPLICATION BLUEPRINT

================================================================================ PART 1: APPLICATION FUNCTIONS The app has 10 core functions:

ASSIGNMENT GRADING Location: Home page → "Grading" tab (default landing tab) Purpose: Grade a single student submission against an assignment prompt and (optional) rubric Inputs: Assignment Prompt (Box A), Grading Instructions (Box B, optional), Student Submission (Box C) Flow: Provider + model + academic level selected → Submit → AI returns grade + structured feedback → results stored Endpoint: POST /api/grade (main path), POST /api/regenerate-feedback (revision with professor notes) Output: Numeric grade, structured feedback, optional charts, persisted to grading_results table

PERFECT ASSIGNMENT GENERATION Location: Home page → "Generate Perfect Answer" panel Purpose: Produce a model A-level answer to the assignment prompt for calibration / comparison Flow: Reads Assignment Prompt → AI generates full exemplar → exemplar stored, displayed beside student work Endpoint: POST /api/generate-perfect-answer, POST /api/generate-perfect-answer-chunked (large prompts), POST /api/regenerate-perfect-answer Output: Exemplar answer (saved to exemplars table) usable as send-target for Compare or Humanizer

ASSIGNMENT COMPARISON Location: Home page → "Compare Assignments" tab Purpose: Side-by-side comparison of a student submission against the perfect answer or any other reference text Flow: Two text inputs → AI scores alignment, identifies gaps, returns comparative feedback Endpoint: POST /api/compare-assignments Output: Per-section delta report, gap analysis, alignment score

AI DETECTION (GPTZero) Location: Inline on Student Submission Box; also surfaced in Humanizer pipeline Purpose: Detect AI-generated content before grading Flow: Text → GPTZero API → returns aiScore (0-100) + classification (HUMAN_ONLY / MIXED / AI_ONLY) Endpoint: POST /api/detect-ai, POST /api/ai-rewriter/analyze-text Output: AI probability + classification badge displayed on submission

AI TEXT REWRITER (HUMANIZER) Location: Dedicated panel — three-box workflow (A=Input, B=Style sample, C=Output) Purpose: Rewrite AI-flagged text to read as human-authored without altering meaning Boxes: A (input text), B (optional style sample for tone match), C (rewritten output) Inter-component sends: C → Student Submission; Chat AI → Box A; Perfect Answer → Box A Endpoint: POST /api/ai-rewriter/upload, POST /api/ai-rewriter/analyze-text, POST /api/ai-rewriter/rewrite, GET /api/ai-rewriter/style-samples, GET /api/ai-rewriter/instruction-presets, POST /api/ai-rewriter/download/:format Output: Rewritten text + before/after GPTZero scores, exportable to TXT/DOCX/PDF

CHAT WITH AI (Assignment Discussion) Location: Home page → Chat panel Purpose: Conversational interface for asking questions about an assignment / submission / grade Providers: All 4 (SHEN 1-6 selectable) Send-targets: Chat response → Student Submission OR Humanizer Box A Endpoint: POST /api/chat-with-ai Output: Conversational response, with one-click send to grading inputs

DOCUMENT / MATH PROCESSING Location: File-upload affordances on all three text boxes Purpose: Accept PDF/DOCX/TXT/image submissions and extract gradable text OCR: Tesseract.js (general), Mathpix (mathematical notation) Endpoint: POST /api/mathpix-ocr, POST /api/format-math, POST /api/assignments/:assignmentId/attachments, GET /api/assignments/:assignmentId/attachments, DELETE /api/attachments/:attachmentId Output: Extracted text inserted into target box, attachments persisted

GRAPH / CHART GENERATION Location: Grading results panel Purpose: Auto-generate performance / breakdown charts from grading output Endpoint: POST /api/generate-graph Output: Inline chart for visual feedback

EMAIL DELIVERY Location: Grading results panel → "Send Email" modal Purpose: Deliver finalized feedback to student via SendGrid Endpoint: POST /api/send-email Output: Status + delivery confirmation, emailSent flag set on grading_results row

SYSTEM DIAGNOSTIC Location: "DIAGNOSTIC" tab (green button in top header, every page) Purpose: Beta-test harness — verify all APIs, DB, and formal CRUD/HTTP flow without grading content judgement Checks (28 total): env vars (8), PostgreSQL connect, users table reach, 6 external API pings (OpenAI / Anthropic / DeepSeek / Perplexity / GPTZero / Mathpix), full CRUD flow (assignment + submission + grading_result, create→read→update→list→delete in FK-safe order), 3 live HTTP route smoke tests (/api/student-grades, /api/detect-ai, /api/grade) Endpoint: POST /api/diagnostic/run Output: Per-check PASS/FAIL/SKIP with latency in ms, overall summary card

================================================================================ PART 6: GRADING SYSTEM

GRADE WEIGHTING (cross-provider system prompt baseline) Correctness 90% (primary — content accuracy, reasoning, factual claims) Style 10% (capped — clarity, organization, tone) Rule: do NOT deduct for stylistic preference unless clarity is compromised.

ACADEMIC LEVEL CALIBRATION (per generateGradingPrompt() in client/src/lib/llm-service.ts) k_12 Generous, encouraging tone; basic conventions community_college Foundational expectations; tolerant of mechanics college_freshman Intro college standards; cite & structure college_sophomore Building disciplinary terminology; stronger argument college_junior Major-specific conventions expected college_senior Capstone-level rigor; thesis + evidence + critique undergraduate_regular Standard undergraduate calibration (DEFAULT) undergraduate_advanced Honors / advanced section calibration graduate Specialist vocabulary; theoretical grounding phd Research-grade rigor; novel contribution expected

FALLBACK BEHAVIOR (grading instructions optional) Layer 1 (client, home.tsx:478): finalGradingText = gradingText || "Please grade this submission using standard academic criteria. Provide a numeric grade and detailed feedback." Layer 2 (server, routes.ts:1940): ${gradingText || 'Standard academic evaluation criteria'} → Submissions with an empty Grading Instructions box always get a deterministic standard-academic rubric.

CHUNKING (very large submissions) needsChunking(provider, assignmentText, gradingText, studentText) decides per provider's context window. processWithChunking() splits at word boundaries, preserves assignment context in every chunk, and reassembles per-chunk feedback into a single coherent grade.

REGENERATION (professor revision loop) POST /api/regenerate-feedback accepts the original grading payload PLUS a professor instructions string, and re-grades with those revisions injected into the prompt. Updates the existing grading_results row.

AI-DETECTION GATING Unregistered users: GPTZero disabled entirely (no UI, no backend calls) Registered users: GPTZero runs on Student Submission text → aiProbability + isAIGenerated stored on submissions row; surfaced as a badge in the UI.

================================================================================ PART 10: KEY ARCHITECTURAL DECISIONS

LAZY-INITIALIZED SDK CLIENTS OpenAI, Anthropic, and PayPal clients are wrapped in lazy getters (getOpenAI(), getAnthropic(), ensurePaypal()). Missing env vars no longer crash the server at module load; they throw a clear error only when the feature is actually invoked.

OPTIONAL GRADING INSTRUCTIONS WITH TWO-LAYER FALLBACK The Grading Instructions box (Box B) is UI-labeled "(Optional)". When empty, the client substitutes a standard-academic rubric before POSTing, and the server independently substitutes one in the prompt builder. The app always grades against deterministic criteria even with zero user input.

ACADEMIC LEVEL CALIBRATION ACROSS 10 LEVELS generateGradingPrompt() switches on GradeLevel union: K-12, Community College, College Freshman/Sophomore/Junior/Senior, Undergraduate Standard, Undergraduate Advanced, Graduate, PhD. Each branch injects distinct calibration text into every provider's system prompt — same model, different rigor.

MULTI-PROVIDER UNIFIED INTERFACE All four providers (SHEN 1 OpenAI, SHEN 3 Anthropic, SHEN 5 DeepSeek, SHEN 6 Perplexity) implement the same LLMGradingRequest shape: assignmentText, gradingText, studentText, model, temperature. Switching providers requires zero downstream changes.

TOKEN-AWARE CHUNKING FOR 400K+ WORD DOCS needsChunking() decides per-provider; processWithChunking() splits on word boundaries, preserves assignment context in every chunk, then reassembles per-chunk feedback. Single grade output regardless of input size.

SELF-DIAGNOSING APP (BETA TEST HARNESS) The /api/diagnostic/run endpoint is the canonical health-check. It exercises every external dependency live AND the app's own HTTP surface, AND runs a complete CRUD round-trip in FK-safe order with full cleanup. 28 checks, typically ~5 seconds, surfaces real bugs (stale model IDs, missing tables, auth misconfigurations) before users do.

DATABASE: EXTERNAL NEON SERVERLESS POSTGRES Uses @neondatabase/serverless with WebSocket transport (ws). All connections go through DATABASE_URL. Schema managed with Drizzle ORM + drizzle-kit push --force for migrations. Tables persist across app restarts.

PERMANENT STORAGE FROM DAY 1 storage.ts uses DatabaseStorage (not MemStorage). All assignments, submissions, grading results, exemplars, attachments, and rewrite sessions survive restarts. User isolation via user_id FK.

INTER-COMPONENT SEND BUTTONS Workflow handoff is one-click throughout: Chat AI → Student Submission Chat AI → Humanizer Box A Perfect Answer → Humanizer Box A Perfect Answer → Student Submission Humanizer Box C → Student Submission No copy/paste required between modules.

KEYBOARD SHORTCUTS ON EVERY INPUT Ctrl+Enter on Student Submission → Grade (min 50 chars) Ctrl+Enter on Assignment Prompt → Move to next field Ctrl+Enter on Grading Instructions → Move to next field Ctrl+Enter on Perfect Generator → Generate Enter on Login / Register forms → Submit Enter in Chat → Send (Shift+Enter for newline)

AI-DETECTION FOR REGISTERED USERS ONLY GPTZero is completely disabled for unregistered users (no UI elements, no backend calls). Eliminates surprise costs and gates an upsell surface.

AUTH DEFERRED — DIAGNOSTIC FIRST Auth (login/register/PayPal/Stripe) is intentionally stubbed for the beta phase. The DIAGNOSTIC tab is unauthenticated so the app can be exercised end-to-end without any user account. Auth + payment hardening is a post-beta task.
